# E语言（E文本）规则说明

本文档结合 `doc/` 目录下的规范资料、以及当前项目中的解析实现整理而成，目标是说明 E 语言 / E文本在本项目里的实际使用规则。

## 1. 基本概念

E 语言可以理解为一种“按表组织、按行书写”的结构化文本格式。它的外层看起来像标签，内层是分行记录的数据体。

在本项目中：

- 一个顶层标签通常对应一张表
- 顶层标签名对应 `ETable.tableName`
- 顶层标签的 `date` 属性对应 `ETable.date`
- 标签正文对应表体数据

典型文件结构大致如下：

```text
<表名 date="2024-01-01">
@ ...
# ...
# ...
</表名>
```

## 2. 文件级规则

### 2.1 编码

当前实现会优先识别文件编码，常见情况是：

- GBK
- UTF-8

如果文件带有 UTF-8 BOM，解析器会按 UTF-8 处理。

### 2.2 注释与头部

文件中支持跳过一些非正文内容：

- `//` 风格注释行
- `<! ... >` 之类的文件头内容

这些内容不会参与表体解析。

### 2.3 顶层节点

文件通常包含一个或多个顶层节点，每个顶层节点就是一张 E 表。

顶层节点规则：

- 节点名是表名
- `date` 属性是日期信息
- 节点文本是表体内容

如果节点名里带有 `::`，当前实现会取 `::` 前面的部分作为表名。

## 3. 表体总规则

表体按行解析，第一行的前缀决定表格式。

### 3.1 三种表格式

| 首行前缀 | 类型 | 说明 |
| --- | --- | --- |
| `@` | 横表式 | 一行定义列，后续行是数据 |
| `@@` | 单列式 | 逐条记录按属性展开 |
| `@#` | 多列式 | 按列块组织数据 |

所有格式都依赖“按空白切分字段”的规则，但会保留引号包裹的整段文本。

## 4. 字段切分规则

字段切分时遵循以下原则：

1. 连续空白会被当成分隔符
2. 单引号或双引号包裹的内容会作为一个整体
3. 引号内部允许出现空格
4. 未加引号的内容按空白切分

示例：

```text
2 发生时间 '2011-11-03 00:00:02.0'
```

切分后会得到类似：

```text
["2", "发生时间", "2011-11-03 00:00:02.0"]
```

## 5. 横表式（`@`）

横表式是最常见的一种表格式。

### 5.1 结构

第一行以 `@` 开头，后面跟列名。

后续可选行包括：

- `%` 行：类型
- `$` 行：单位
- `:` 行：限值
- `#` 行：数据

### 5.2 规则

1. 第一行解析为列名数组
2. `%` 行写入类型数组
3. `$` 行写入单位数组
4. `:` 行写入限值数组
5. `#` 行写入数据行

### 5.3 含义

横表式适合“列固定、行变化”的结构。

例如：

```text
@ 编号 名称 值
% int string decimal
$ - - MW
# 1 线路A 12.5
# 2 线路B 18.0
```

## 6. 多列式（`@#`）

多列式用于按“列块”组织数据。

### 6.1 结构

第一行以 `@#` 开头。

后续每一行以 `#` 开头，表示一列或一组属性。

### 6.2 规则

1. 只处理 `#` 开头的行
2. 去掉 `#` 后按字段切分
3. 每行的第 2 个字段作为列名
4. 从第 3 个字段开始作为该列的数据序列
5. 多行按位置拼成最终二维数据

### 6.3 示例

```text
@# ...
# 1 列A A1 A2
# 2 列B B1 B2
```

解析后可理解为：

```text
columnNames = ["列A", "列B"]
datas = [
  ["A1", "B1"],
  ["A2", "B2"]
]
```

## 7. 单列式（`@@`）

单列式用于把一条记录中的多个属性逐项展开。

### 7.1 结构

第一行以 `@@` 开头。

后续每条记录由 `#` 开头。

### 7.2 规则

1. 第一行给出表头定义
2. 从第二行开始处理 `#` 行
3. 去掉 `#` 后按字段切分
4. 根据表头字段数量决定属性名和值的位置
5. 使用有序 Map 暂存一条记录
6. 如果后续行出现重复属性名，说明上一条记录结束
7. 将上一条记录写入数据列表
8. 最后一条记录在循环结束后补写

### 7.3 特点

单列式更适合“同一条记录里属性很多、每个属性单独列出”的场景。

## 8. E表与 Java 类的映射

本项目里，E 文本先被解析成 `ETable`，再转换成 Java 业务对象。

### 8.1 表名映射

Java 类上通常会使用表注解来声明它对应哪张表。

规则是：

- Java 类上的表名注解值
- 必须与 `ETable.tableName` 一致

否则会判定为表不匹配。

### 8.2 字段映射

字段映射依赖列名注解。

规则是：

- E 文本里的列名通常是中文
- Java 类里的字段名通常是英文驼峰
- 二者通过 `@EColumn("中文列名")` 绑定

例如：

```java
@EColumn("发生时间")
private String occurTime;
```

这里：

- `发生时间` 是 E 文本里的列名
- `occurTime` 是 Java 类里的属性名

### 8.3 映射顺序

解析时通常按列顺序把一行数据灌入对象：

1. 先根据注解找到目标列
2. 再记录列下标
3. 遍历每一行数据
4. 按列下标取值
5. 反射写入字段

## 9. 规则总结

可以把 E 文本的核心规则记成下面几句：

1. 一个顶层标签对应一张表
2. 表体第一行决定格式
3. 行内容按空白切分，但引号中的文本整体保留
4. `@` 是横表式，`@@` 是单列式，`@#` 是多列式
5. 中文列名通过注解映射到 Java 英文字段名
6. 表名通过注解映射到 Java 类

## 10. 对当前项目的补充说明

当前实现不是完整 XML 解析器，而是针对项目中的 E 文本格式做的轻量自定义解析。

从使用角度看，它更像是：

- 先把文本包装成 DOM
- 再把 DOM 转成 `ETable`
- 再按注解转成业务对象

如果后续要扩展，最值得关注的点是：

- 是否需要补强标签属性读取
- 是否需要把普通空格也纳入切分规则
- 是否需要支持更多字段类型转换
- 是否需要支持更复杂的嵌套结构
